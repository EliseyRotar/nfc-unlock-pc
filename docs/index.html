NFC Unlock PC

Tap your phone on a USB ACR122U reader and your PC unlocks — Windows or Linux, no sticker required. On Linux it can unlock without ever typing a password.

MIT Licensed Python Windows 10/11 & Linux Convenience tool, not FIDO2

What it does

Every phone made in the last decade has an NFC radio capable of Host Card Emulation (HCE) — acting as a card, not just reading one. The companion Android app uses that to make your phone answer a USB ACR122U NFC reader with a private 16-byte token, generated once on-device. No sticker, no extra hardware — your phone is the credential.

A small background process on your PC watches the reader. When it sees your enrolled token (or a physical NTAG/MIFARE tag, if you'd rather use one), it unlocks your session — using whichever of the two tiers below is available on your OS.

Two tiers of unlock

Tier 1 — Linux, PAM: no password ever typed

scripts/setup_pam_linux.sh <service> adds one line to /etc/pam.d/<service>: auth sufficient pam_exec.so quiet python scripts/pam_nfc_check.py. PAM runs that script, which exits 0 if your enrolled phone is on the reader (within 5s) — and PAM treats that as a successful login, no password prompt at all. If it exits non-zero, PAM falls straight through to your normal password check, completely unchanged. You cannot be locked out by this: worst case it behaves exactly as if it were never installed.

This is the same external-helper pattern used by SiRFIDaL and pam_nfc — proven prior art, not a novel hack.

Tier 2 — typed password (Linux fallback, or Windows)

Your account password lives in your OS's secure credential store (keyring — Windows Credential Manager / Linux Secret Service / KWallet), never in plaintext. On tap, it's typed into the lock screen for you.

Windows runs the service as SYSTEM via a Scheduled Task so it can attach to the isolated Winlogon secure desktop and call SendInput — ordinary processes can't do this, which is also why Windows has no Tier-1 equivalent yet (see Roadmap). Linux runs a normal systemd --user service using xdotool (X11) or wtype (Wayland), since lockers run inside your own session.

How identification works

┌──────────────────────┐ tap ┌────────────┐ PC/SC (pyscard) ┌────────────────────┐ │ Phone: HCE companion │ ──────────────────▶│ ACR122U │ ───────────────────▶│ reader.py │ │ app (or NTAG sticker)│ SELECT AID │ USB reader│ get_identifier() │ get_identifier() │ │ │ F0AC1DC0DE0001 └────────────┘ └─────────┬──────────┘ └──────────────────────┘ │ == enrolled? ┌──────────────────────────────┴───────────────────────────┐ ▼ ▼ Linux: pam_nfc_check.py exit 0 Linux Tier 2 / Windows: → PAM accepts, locker unlocks type stored password + Enter (nothing typed)

Physical tags answer FF CA 00 00 00 with their UID. Phones running the HCE app don't have a stable UID (Android randomizes it per-tap for privacy), so the app instead registers AID F0AC1DC0DE0001 and answers SELECT with a fixed token generated on first launch. reader.get_identifier() tries a UID read first, then that AID — so enrollment and unlock code don't care which kind of "tag" you're using.

Get started

  1. Plug in the ACR122U (install the official ACS driver on Windows).
  2. git clone https://github.com/EliseyRotar/nfc-unlock-pc.git && cd nfc-unlock-pc && python install.py
  3. Build android-hce-app/ in Android Studio and install it on your phone (or write an NTAG sticker instead).
  4. python src/main.py enroll — tap your phone, optionally store your password.
  5. Linux: sudo bash scripts/setup_pam_linux.sh <pam-service> for Tier 1 (recommended), or bash scripts/install_service_linux.sh for Tier 2.
    Windows: scripts\install_task.ps1 as Administrator.
  6. Lock your PC, tap your phone — done.

Full setup instructions, troubleshooting and security notes are in the project README.

Project layout

reader.py

Talks to the ACR122U via PC/SC (pyscard). get_identifier() reads a physical tag's UID or a phone's HCE token (AID F0AC1DC0DE0001).

android-hce-app/

Minimal HCE app: generates a token on first launch, answers the AID SELECT with it. That's the whole app.

pam_nfc_check.py + setup_pam_linux.sh

Tier 1: the PAM helper and the script that wires it into /etc/pam.d/<service> as sufficient.

config.py

Stores the enrolled identifier in config.json; password (optional) goes through keyring into the OS credential store.

enroll.py

One-time wizard: tap phone/tag, optionally save a password, prints the right next command for your OS.

unlocker_windows.py / unlocker_linux.py

Tier 2 typing backends — SendInput to Winlogon on Windows, xdotool/wtype on Linux.

Security notes

This is a convenience tool, not a certified authentication system — anyone who presents an enrolled phone/tag to the reader can unlock your session, the same threat model as a physical access card. The HCE token is a 16-byte random value tied to the phone (not a cloneable UID), making it the strongest identifier this project supports. Tier 1's PAM line is sufficient and inserted first — if the check ever fails or the reader is unplugged, your normal password prompt appears exactly as before. Tier 2 passwords sit in your OS's secure credential store, with the same exposure as any saved Windows/browser credential.

Roadmap: Windows without typing

A true Tier-1 equivalent on Windows means a custom Credential Provider (ICredentialProvider COM interfaces) — the same plug-in point Windows Hello and smart-card logon use. Research for this project found no actively-maintained open-source reference implementation for an NFC-based Credential Provider; Rohos Logon Key is the closest commercial equivalent. Building one is a real COM/C++ project and is tracked as future work, not promised functionality — Windows currently uses Tier 2 (typed password via SYSTEM + Winlogon) only.
